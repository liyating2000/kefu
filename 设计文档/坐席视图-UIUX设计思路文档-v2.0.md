- 项目名称：个人门户坐席视图 2.0
- 文档类型：UI/UX 设计思路与落地规范
- 适用范围：个人门户首页中的坐席视图
- 交付用途：设计评审、开发对接、联调验收、后续迭代参考

## 1. 设计定位与核心目标

### 1.1 项目类型定义

本项目属于典型的企业内部工作台型 Dashboard，服务对象是客服坐席。它不是展示品牌形象的营销首页，也不是单纯的信息看板，而是一个围绕“高频判断、高频动作、持续反馈”构建的业务工作入口。

### 1.2 设计风格定位

坐席视图 2.0 的设计风格定位为：

- 轻玻璃感企业工作台风格
- 高密度但不拥挤的信息驾驶舱
- 强秩序感、弱装饰性的专业界面
- 兼顾业务效率与团队激励的运营门户

风格关键词如下：

- 专业
- 清晰
- 稳定
- 现代
- 可扩展

### 1.3 目标用户

核心用户为一线客服坐席，细分特征如下：

- 高频使用系统，每日长时间停留在工作台内
- 需要在极短时间内建立当日工作状态判断
- 需要从多个模块中快速定位待办、风险与协同入口
- 对页面的“扫描效率”敏感，高于对页面“装饰性”的关注
- 在热线与在线两类接待模式之间切换，需要统一但可区分的视觉逻辑

### 1.4 设计核心诉求

坐席视图 2.0 的核心诉求不是“内容更多”，而是“更快做判断、更少找入口、更稳完成动作”。因此本版本重点解决以下问题：

- 首屏进入后的信息建立时间过长
- 模块过于并列，主次不清
- 高优先级工作入口没有形成连续路径
- 提醒信息与激励信息混在一起，影响主工作链路
- 同一页面中缺少统一的组件语法与视觉节奏

### 1.5 核心目标

1. 让坐席在 3 秒内建立当日工作状态的整体认知。
2. 让用户沿从左到右的浏览路径，自然完成“看指标、看动作、进处理”的扫描流程。
3. 用渐进式披露控制复杂度，先展示结论，再展示说明，再提供入口。
4. 用统一的卡片、标签、分段控件和动作入口形成稳定的设计系统。
5. 为后续扩展到更多角色门户、更多业务线模块预留通用框架。

## 2. 整体设计系统

### 2.1 色彩系统

本项目的色彩系统采用“品牌主色 + 功能辅助色 + 大面积中性色”的三层结构，目的是保证专业感、可读性和状态识别效率。

#### 2.1.1 主色

| 色彩角色 | 色值 | 使用场景 | 规范说明 |
| --- | --- | --- | --- |
| 主品牌色 | `#12B89F` | 关键强调、激活态、正向提示、强调入口 | 不大面积铺底，优先用于激活态和关键强调 |
| 主品牌深色 | `#0F9F89` | 主色文字、链接、次级强调动作 | 用于文字型 CTA，保证克制和专业 |
| 主色浅底 | `#ECFBF8` | 激活背景、轻提示底色 | 仅作功能性浅底，不作为整页背景 |

#### 2.1.2 辅助色

| 色彩角色 | 色值 | 使用场景 | 规范说明 |
| --- | --- | --- | --- |
| 效率蓝 | `#4F7CFF` | 在线/效率/信息型模块辅助强调 | 用于效率类视觉辅助，避免与主色争抢主导权 |
| 质量橙 | `#F59E0B` | 质量预警、需跟进提醒、重点关注标签 | 用于提醒，不直接表示错误 |
| 风险红 | `#F43F5E` | 明确风险、异常、超阈值状态 | 必须与文字或图标共同出现，不单独依赖颜色 |
| 激励金 | `#FFB400` | 排名、勋章、激励模块 | 仅用于激励场景，避免进入业务指标主系统 |
| 信息青 | `#38BDF8` | 渐变头像、说明性增强 | 用作视觉装饰与辅助，不承担核心状态含义 |

#### 2.1.3 中性色

| 色彩角色 | 色值 | 使用场景 | 规范说明 |
| --- | --- | --- | --- |
| 页面主文字 | `#0F172A` | 主标题、核心数值、重要标签 | 高对比、保证首屏扫描效率 |
| 次级文字 | `#475569` | 说明文案、摘要、辅助说明 | 用于大部分正文说明 |
| 弱提示文字 | `#94A3B8` | 辅助标签、次级时间信息、元信息 | 不可承载主要业务信息 |
| 页背景 | `#F4F8FC` | 页面基础背景 | 提供柔和、稳定的企业工作氛围 |
| 浅背景层 | `#F8FBFF` | 区域过渡、上层背景渐变起点 | 作为层级过渡，不建议承载复杂信息 |
| 边框基色 | `#CBD5E1` | 组件边框、分隔结构、浅状态容器 | 透明度可根据层级微调 |
| 卡片白底 | `rgba(255,255,255,0.94)` | 主信息卡片背景 | 保证内容承载稳定性 |

#### 2.1.4 色彩使用原则

1. 主色只负责“当前操作重点”，不负责所有显眼内容。
2. 风险、预警、正常三类状态必须稳定映射到红、橙、绿系。
3. 大面积区域以中性色为主，彩色只出现在功能节点。
4. 激励模块可使用金橙系，但不得干扰业务数据主阅读路径。
5. 文本与背景对比需满足 WCAG AA，正文至少 4.5:1。

### 2.2 字体系统

当前字体栈采用：

`Aptos, Segoe UI Variable Display, Microsoft YaHei UI, PingFang SC, Hiragino Sans GB, sans-serif`

设计策略如下：

- 英文和数字优先使用现代办公风格字形，保证数据感和专业感
- 中文优先使用系统可用的高可读性无衬线字体
- 多端兼容时优先系统字体，减少加载风险

#### 2.2.1 字号层级

| 层级 | 建议字号 | 字重 | 行高 | 使用场景 |
| --- | --- | --- | --- | --- |
| Display 级 | 26px | 600 | 1.25 | 页面欢迎语、页面主标题 |
| H1 级 | 22px | 600 | 1.3 | 一级区块标题 |
| H2 级 | 20px | 600 | 1.35 | 次级模块标题 |
| H3 级 | 18px | 600 | 1.4 | 卡片标题、模块标题 |
| Title 级 | 15px | 600 | 1.4 | 业务卡片标题、人员姓名 |
| Body 主正文 | 14px | 400/500 | 1.6 | 页面正文、摘要说明 |
| Body 辅助正文 | 13px | 400/500 | 1.55 | 标签说明、模块内部描述 |
| Meta 信息 | 12px | 400/500 | 1.5 | 时间、辅助状态、场景标签 |
| Eyebrow/Overline | 11px | 500 | 1.4 | 英文眉标、分类提示 |

#### 2.2.2 字重规则

- `600`：模块标题、关键数字、关键标签
- `500`：按钮文本、状态标签、段落中需要强调的信息
- `400`：说明文案、摘要文案、元信息

#### 2.2.3 行高规则

- 大标题：`1.25 - 1.35`
- 普通正文：`1.55 - 1.6`
- 小字号说明：`1.45 - 1.5`

#### 2.2.4 使用规范

1. 标题层级不超过 3 级，避免页面出现过多抢眼文本。
2. 数值优先使用更高字重和更紧凑字距，提升数据识别速度。
3. 说明文案不超过两行时优先控制长度而非缩小字号。
4. 小于 12px 的正文文本禁止用于承载核心信息。

### 2.3 间距规范

本项目采用 4px 基础栅格和 8px 主节奏组合。

#### 2.3.1 基础单位

- 基础单位：`4px`
- 主节奏单位：`8px`

#### 2.3.2 推荐间距梯度

| 间距等级 | 数值 | 使用场景 |
| --- | --- | --- |
| XS | 4px | 图标与文字微间距 |
| S | 8px | 同一组件内弱关联元素 |
| M | 12px | 轻量信息块内间距 |
| L | 16px | 同一卡片内模块间距 |
| XL | 20px | 卡片内主内容间距 |
| XXL | 24px | 卡片边距、模块边距 |
| Section | 32px | 大区块间距 |

#### 2.3.3 当前页面的间距落点

- 页面主容器内边距：`24px`
- 卡片内边距：`20px`
- 卡片圆角：`24px`
- 行内按钮内边距：`12px - 14px`
- 模块间主间距：`16px - 24px`
- 列表卡片间距：`12px`

#### 2.3.4 间距使用规则

1. 同级内容采用同一间距，避免视觉节奏失真。
2. 模块内间距永远小于模块间间距。
3. 说明文案与主标题间距应小于标题与下一区块间距。
4. 操作区与说明区必须用间距分层，不依赖分割线硬切。

## 3. 页面结构与布局逻辑

### 3.1 整体布局框架

坐席视图 2.0 采用“顶部欢迎区 + 主工作区”的二层结构，主工作区在桌面端进一步分为“左主列 + 右辅助列”。

#### 3.1.1 顶部欢迎区

包含以下内容：

- 欢迎语
- 视图切换
- 热线/在线切换

这一层的目标不是承载业务内容，而是建立身份、场景和入口切换。

#### 3.1.2 主工作区

桌面端采用左右双列结构：

- 左主列：承载核心指标与处理入口
- 右辅助列：承载提醒集合与激励信息

当前列宽比例策略可描述为：

- 左列约 76%
- 右列约 24%

这是一个明显偏向主工作信息的布局，符合坐席首屏“先工作、后提醒、再激励”的优先级。

### 3.2 模块划分

当前坐席视图 2.0 由以下主要模块构成：

1. 顶部欢迎区
2. 通道切换区
3. 指标总览区
4. 工单协同
5. 知识库推荐
6. 今日动作
7. 明星员工

### 3.3 视觉层级设计思路

页面视觉层级遵循“先判断、再动作、后激励”的顺序。

第一层：

- 欢迎语
- 热线/在线模式切换
- 指标总览

第二层：

- 工单协同
- 知识库推荐
- 今日动作

第三层：

- 明星员工

这种层级有两个好处：

- 用户进入页面时，先看“现在怎么样”
- 再看“我接下来做什么”
- 最后看“团队激励与氛围”

### 3.4 布局合理性说明

#### 3.4.1 欢迎语放在左上角的原因

- 左上角是用户进入页面后的第一视觉锚点
- 放欢迎语可以帮助用户快速建立“我在什么页面、当前角色是谁”的认知
- 对于高频使用的工作台，欢迎语不只是情绪表达，更是身份确认与心理启动
- 结合用户从左到右的阅读习惯，先看到身份与今日工作主题，再进入数据扫描更自然

#### 3.4.2 视图切换与热线/在线切换放在右上角的原因

- 这是场景切换控件，不属于主内容，应与内容区保持分离
- 右上角适合放全局控制，符合多数企业后台和工作台产品习惯
- 视图切换与通道切换并列放置，可形成统一的控制带
- 使用分段控件而非下拉框，可降低切换成本，提高状态可见性

#### 3.4.3 指标总览区放在左上主列的原因

- 指标区是首屏最核心的业务判断层
- 坐席进入页面后首要任务是判断当天当前状态，而不是立刻进入具体工单
- 指标按照主题拆为“通话效率 / 服务质量 / 状态时长”或“响应效率 / 服务质量 / 时长结构”，比将多个指标平铺更利于快速扫描
- 每组指标内部再拆成 4 个小单元，符合渐进式披露原则：先看模块名，再看关键值，再看说明，再决定是否下钻

#### 3.4.4 工单协同放在左列第二层靠左的原因

- 工单协同是最直接的执行入口
- 在大多数坐席场景中，用户完成状态判断后，下一步最可能进入工单处理
- 放在左主列第二层，既承接指标区，又保持与知识库推荐并列
- 从左到右的浏览路径中，用户先看左侧执行，再看右侧支持，符合业务流

#### 3.4.5 知识库推荐放在工单协同右侧的原因

- 知识库本质上是“支持决策和标准化表达”的辅助模块
- 它应贴近执行入口，但不应高于执行入口
- 与工单协同并列，能形成“执行 + 支持”的双核结构
- 用户在处理工单时可顺势查看推荐知识，无需跨列跳读

#### 3.4.6 今日动作放在右上辅助列的原因

- 今日动作属于提醒与轻任务集合
- 其优先级高于激励信息，但低于主指标判断
- 放在右上辅助列可保证可见性，同时不挤占左列核心工作区
- 将“今日必看”并入同一卡片，是对提醒信息的归并处理，减少模块数量，降低右栏噪音

#### 3.4.7 明星员工放在右下角的原因

- 明星员工属于激励型信息
- 它对团队氛围有价值，但不应在工作流早期占据主要视觉资源
- 放在页面右下作为收束信息，可以在完成主任务扫描后自然看到
- 从心理体验上，它承担的是“工作后的情绪补充”，而非“工作前的判断依据”

### 3.5 渐进式披露原则的应用

本页面严格遵循渐进式披露原则，将复杂信息拆分成多层呈现：

第一层：

- 模块名
- 标题
- 当前模式

第二层：

- 关键指标值
- 关键动作项
- 状态标签

第三层：

- 元信息说明
- 处理时限
- 命中率
- 场景说明

第四层：

- 跳转行为
- 下钻页
- 排名详情
- 工单详情

这样做的好处是：

- 用户不需要一次理解所有信息
- 能先建立整体判断，再决定是否进入细节
- 有效避免首屏信息过载

### 3.6 从左到右视觉浏览路径原则的应用

页面设计刻意贴合用户自然阅读路径：

1. 左上先建立角色与工作语境
2. 右上确认当前视角与服务模式
3. 回到左侧主列扫描核心指标
4. 向下进入工单协同与知识支持
5. 最后移动到右侧查看提醒与激励

这条路径的本质是：

- 先全局
- 再判断
- 再执行
- 再提醒
- 再激励

这比“所有模块平均陈列”的效率更高，也更符合工作型页面的使用方式。

## 4. 组件设计逻辑

### 4.1 欢迎头部组件

#### 设计规范

- 左侧为欢迎语与页面说明
- 右侧为视图切换与模式切换
- 标题使用 26px、600 字重
- 说明文案控制在两行以内

#### 使用场景

- 页面首次进入
- 角色身份确认
- 日常工作开始阶段

### 4.2 分段切换组件

适用于：

- 坐席视图 / 管理员视图切换
- 热线 / 在线切换

#### 样式标准

- 外层：圆角 22px，浅灰边框，半透明白底
- 内层按钮：圆角 18px
- 激活态：白底 + 轻阴影 + 深色文字
- 非激活态：透明底 + 中灰文字

#### 交互状态

| 状态 | 表现 |
| --- | --- |
| 默认 | 中灰文字、浅底 |
| Hover | 文字加深 |
| Active | 白底、阴影、文字加深 |
| Disabled | 透明度降低，不建议在本场景高频出现 |

### 4.3 指标模块卡片

#### 设计规范

- 每组为一个主题卡
- 组内指标拆分为多个小卡
- 模块包含：眉标、标题、描述、指标栅格
- 颜色仅作为主题辅助，不覆盖内容可读性

#### 使用场景

- 通话效率
- 服务质量
- 状态时长
- 响应效率
- 时长结构

#### 交互状态

- 可点击项显示弱操作提示图标
- Hover 时卡片轻微上浮
- 边框和阴影轻微增强

### 4.4 工单协同卡片

#### 设计规范

- 单条工单使用独立卡片
- 卡片内容包含：标题、状态、优先级、责任人、截止时间、摘要、动作按钮
- 状态通过胶囊标签体现

#### 使用场景

- 待升级工单
- 待回访工单
- 处理中工单

#### 价值

- 缩短从判断到执行的路径
- 让待办工单更像任务，而不是列表记录

### 4.5 知识库推荐卡片

#### 设计规范

- 单条知识卡包含：标题、命中率、场景、摘要、标签
- 命中率作为右上高价值信息
- 标签用于提供更快的二次筛选感知

#### 使用场景

- 标准话术
- SOP 指引
- 退款说明模板
- 物流说明模板

### 4.6 今日动作卡片

#### 设计规范

- 一个模块内包含“今日必看”卡片和动作列表卡片
- “今日必看”与动作列表采用统一列表卡样式
- 动作项结构为：标题、值、说明
- 通过色彩区分优先级，不用复杂装饰

#### 使用场景

- 待回访工单
- 待复盘录音
- 班次提醒
- 明日班次
- 今日必看公告

### 4.7 明星员工卡片

#### 设计规范

- 上方为榜单类型切换
- 中部为前三名奖章化展示
- 底部为“我的排名”信息

#### 使用场景

- 沟通量排行
- 满意度排行

#### 设计目的

- 提升页面情绪温度
- 提供团队竞争与激励氛围
- 作为首屏信息收尾，增强页面完整度

## 5. 交互设计思路

### 5.1 页面跳转逻辑

坐席视图的页面跳转遵循“同屏判断，按需下钻”的原则。

主跳转路径如下：

1. 指标卡点击进入质量详情或录音详情
2. 工单协同进入工单管理页
3. 知识库推荐进入知识库
4. 今日必看进入消息服务
5. 明星员工进入排名详情页

### 5.2 微交互效果

当前页面适合采用轻量微交互，而不是强动效：

- Hover 位移：`-2px`
- 阴影增强：弱提升
- 按钮激活：底色切换 + 阴影增强
- 下钻入口：箭头轻微横向位移

### 5.3 Hover / 点击反馈规则

| 元素 | Hover 反馈 | 点击反馈 |
| --- | --- | --- |
| 卡片 | 轻微上浮、边框增强、阴影增强 | 保持结构稳定，避免剧烈位移 |
| 分段按钮 | 文本加深 | 激活态切换到白底 |
| 行内文字按钮 | 文字颜色加深、箭头轻微偏移 | 进入详情页 |
| 标签型卡片 | 边框或浅底增强 | 触发筛选或下钻 |

### 5.4 响应式交互规则

建议按以下断点处理：

- `375px`：移动端单列
- `768px`：平板双段纵排
- `1024px`：桌面基础双列
- `1440px`：主工作分辨率

响应式原则如下：

1. 指标卡从三列降到单列或双列
2. 工单协同与知识库推荐在窄屏下改为上下堆叠
3. 右侧提醒与激励模块在移动端下沉到主列之后
4. 分段按钮保持足够点击热区，最小 44px

## 6. 体验优化点

### 6.1 当前页面已完成的优化方向

1. 将坐席视图标题升级为欢迎语，强化角色与工作语境。
2. 将热线/在线切换统一为可见式分段控件，减少下拉认知成本。
3. 将“每日必看”合并进“今日动作”，减少右栏模块碎片化。
4. 将工单协同与知识库推荐放在同一层级，形成执行与支持组合。
5. 将明星员工保留在右下角，避免干扰主工作流。

### 6.2 仍可继续优化的不足

#### 不足 1：指标数据的强弱对比仍可进一步拉开

问题：

- 部分说明文字与核心值的视觉区分仍偏温和

优化方向：

- 强化数字与说明之间的字重和对比层级
- 对异常项增加更明确的状态文案

#### 不足 2：今日动作尚未形成真正的优先级排序视觉

问题：

- 当前主要靠颜色区分优先级，排序感还可加强

优化方向：

- 增加序号、分组标题或时间约束标签

#### 不足 3：明星员工与业务区之间的边界仍偏弱

问题：

- 激励区虽已下沉，但与上方任务区仍属于同类卡片语言

优化方向：

- 后续可进一步提高激励模块的识别度，如加入更明确的奖励色区分

#### 不足 4：跨页面组件 token 还未完全抽离

问题：

- 当前样式规范已经形成，但尚未完全产品化为全局 token

优化方向：

- 将颜色、阴影、圆角、字号、间距抽离成统一设计 token 与前端变量

### 6.3 优化逻辑总结

本版本优化不是盲目增加视觉变化，而是遵循以下逻辑：

- 优先提升决策效率
- 其次提升操作连续性
- 最后补充页面情绪价值

## 7. 设计原则说明

### 7.1 渐进式披露原则

#### 原则说明

复杂信息不一次性全部展开，而是按重要性和操作路径逐层释放。

#### 本项目中的应用

- 指标模块先显示主题，再显示关键值，再显示说明
- 工单卡先显示标题和状态，再显示摘要和动作
- 今日动作先展示最需要知道的提醒，再展示完整动作列表

### 7.2 用户从左到右的视觉浏览路径原则

#### 原则说明

页面结构应尽量顺应中文用户常见的左到右、从上到下阅读习惯。

#### 本项目中的应用

- 左上欢迎语起始
- 右上完成模式确认
- 左侧主列先承载判断与执行
- 右侧后承载提醒与激励

### 7.3 亲密性原则

#### 原则说明

相关信息应靠近摆放，不相关信息要拉开距离。

#### 本项目中的应用

- 工单标题、状态、优先级、责任人组合在同一卡片头部
- 知识标题、命中率、场景、标签形成单元
- 今日必看与动作列表归并在同一模块

### 7.4 对齐原则

#### 原则说明

元素必须建立清晰的视觉对齐关系，减少界面噪音。

#### 本项目中的应用

- 模块标题统一左对齐
- 指标卡栅格统一排列
- 右侧辅助列采用统一卡片宽度和内边距

### 7.5 重复原则

#### 原则说明

一致的样式重复使用，帮助用户形成稳定的界面认知。

#### 本项目中的应用

- 卡片圆角统一
- 标签形态统一
- 分段按钮统一
- 行内入口统一采用箭头按钮

### 7.6 对比原则

#### 原则说明

通过字号、颜色、阴影、权重差异建立重点。

#### 本项目中的应用

- 核心数值与说明文案形成明显大小对比
- 风险项使用红色或橙色标签突出
- 激活按钮与非激活按钮形成明确对比

### 7.7 视觉层级原则

#### 原则说明

页面必须让用户一眼分辨“先看什么、后看什么”。

#### 本项目中的应用

- 头部欢迎区最先建立语境
- 指标区为一级内容
- 工单、知识、今日动作为二级内容
- 明星员工为三级内容

### 7.8 用户中心原则

#### 原则说明

设计不以“模块齐全”为目标，而以“用户完成任务更顺”为目标。

#### 本项目中的应用

- 将知识库放在工单旁边而不是单独远离
- 将每日必看并入今日动作，降低碎片化
- 将明星员工下沉，避免影响首屏工作流

## 8. 落地注意事项

### 8.1 开发适配要点

1. 所有颜色、圆角、阴影、字号建议抽为 token。
2. 分段控件、卡片、标签、行内动作建议封装为基础组件。
3. 不同场景下的模块内容可以变，但结构骨架保持一致。
4. 热线和在线仅切换数据与文案，不建议改变整体布局框架。

### 8.2 Accessibility 规范

1. 正文文字与背景对比度不低于 4.5:1。
2. 所有按钮热区不低于 `44px × 44px`。
3. 图标按钮需有 `aria-label`。
4. 分段控件需支持键盘切换与焦点态可见。
5. 风险状态不能只靠颜色区分，必须有文字标签。

### 8.3 视觉一致性要求

1. 所有主卡片圆角统一为 `24px`。
2. 所有信息列表卡统一为 `18px - 20px` 圆角。
3. 所有主按钮、分段按钮和轻动作链接都应遵循统一反馈逻辑。
4. 说明文案统一使用次级文字色，避免模块间风格漂移。

### 8.4 性能与实现建议

1. 避免使用大量高成本模糊与阴影叠加。
2. 动效仅使用 `transform` 与 `opacity`。
3. 图像资产尽量本地化或稳定托管，避免随机外链资源影响体验。
4. 首屏信息模块应优先保证可见与可操作，避免异步加载造成布局跳动。

## 9. 补充说明

### 9.1 设计亮点

1. 将传统客服首页重构为真正的工作流入口，而非信息堆叠页。
2. 用左主右辅的结构，建立了明确的业务优先级。
3. 用“工单协同 + 知识库推荐”构建执行与支持的组合关系。
4. 用“今日必看 + 今日动作”形成提醒闭环，减少模块碎片。
5. 用“明星员工”保留团队激励，但不影响主工作链路。

### 9.2 可扩展方向

1. 扩展为多角色门户模板，支持主管、质检、排班等角色复用。
2. 抽离为企业级设计系统页面模板。
3. 增加个性化推荐逻辑，如按坐席能力、班次和业务类型调整模块顺序。
4. 引入更精细的风险优先级机制，如 SLA 超时、重复来访、投诉升级预警。
5. 后续可补充空状态、异常状态、加载状态的完整规范页。

### 9.3 结论

坐席视图 2.0 的设计重点，不是做成“更花哨的一页首页”，而是做成“更高效的一页工作入口”。  
本方案通过统一的设计系统、明确的左右布局逻辑、稳定的组件语言、渐进式披露策略以及贴合用户从左到右浏览习惯的内容组织方式，让坐席能够更快建立判断、更顺完成动作、更稳定处理业务。
