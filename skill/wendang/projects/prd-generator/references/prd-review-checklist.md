# PRD审查清单

本文档定义了PRD文档的专业审查维度和检查项，用于指导智能体以产品经理和开发工程师的双视角进行全面审查。

## 产品经理视角审查

### 业务逻辑审查
- [ ] 业务场景是否完整覆盖用户需求？
- [ ] 用户角色和权限定义是否合理？
- [ ] 业务流程是否闭环，是否存在断点？
- [ ] 异常场景是否充分考虑？
- [ ] 业务规则是否存在冲突或遗漏？
- [ ] 功能优先级是否明确？
- [ ] 是否存在重复或冗余的功能？

### 用户体验审查
- [ ] 交互流程是否符合用户习惯？
- [ ] 操作步骤是否简洁高效？
- [ ] 错误提示是否友好明确？
- [ ] 页面布局是否合理？
- [ ] 字段命名是否易于理解？

### 产品定位审查
- [ ] 核心功能是否突出？
- [ ] 功能边界是否清晰？
- [ ] 是否存在过度设计或功能膨胀？
- [ ] 是否考虑了扩展性？

---

## 开发工程师视角审查

### 技术可行性审查
- [ ] 需求描述是否足够详细，能够指导开发？
- [ ] 数据字段定义是否完整（类型、长度、校验规则）？
- [ ] 接口依赖是否明确？
- [ ] 是否存在技术实现难点或风险？
- [ ] 性能要求是否明确且可量化？
- [ ] 安全要求是否具体可实施？

### 开发细节审查
- [ ] 状态机转换是否清晰完整？
- [ ] 数据校验规则是否覆盖所有场景？
- [ ] 并发场景是否考虑？
- [ ] 数据一致性问题是否说明？
- [ ] 边界条件是否明确？
- [ ] 异常处理机制是否完整？

### 技术架构审查
- [ ] 是否需要新系统或模块？
- [ ] 是否需要数据库表结构变更？
- [ ] 是否需要第三方服务集成？
- [ ] 是否需要特殊的中间件或工具？

## 疑问清单格式

审查完成后，生成结构化的疑问清单：

```markdown
# PRD审查疑问清单

## 审查概述
- 审查时间：{时间}
- 审查角色：产品经理、开发工程师
- 发现问题：{数量}项

## 产品经理视角疑问

### 高优先级
1. **[业务逻辑]** {章节位置}：{疑问描述}
   - 影响范围：{说明影响哪些功能或用户}
   - 建议：{提供可选方案}
   - 待确认：{需要用户回答的问题}

2. **[用户体验]** {章节位置}：{疑问描述}
   - 影响范围：{说明}
   - 建议：{可选方案}
   - 待确认：{问题}

### 中优先级
{同上格式}

## 开发工程师视角疑问

### 高优先级
1. **[技术可行性]** {章节位置}：{疑问描述}
   - 影响范围：{说明影响开发进度或成本}
   - 建议：{技术方案建议}
   - 待确认：{问题}

2. **[数据设计]** {章节位置}：{疑问描述}
   - 影响范围：{说明}
   - 建议：{方案}
   - 待确认：{问题}

### 中优先级
{同上格式}

## 其他建议
{列出优化建议或改进意见}
```

## 使用说明

**审查时机**：PRD文档生成完成后

**审查流程**：
1. 按照产品经理视角清单逐项检查
2. 按照开发工程师视角清单逐项检查
3. 将发现的问题按优先级整理
4. 生成结构化的疑问清单
5. 向用户展示并逐一确认处理方式

**优先级定义**：
- **高优先级**：影响核心功能、存在逻辑冲突、可能导致开发受阻的问题
- **中优先级**：影响用户体验、可能导致理解偏差、需要明确细节的问题
- **低优先级**：优化建议、改进意见、非必须调整的问题

**审查原则**：
- 不猜测或假设，有疑问必须提出
- 每个疑问都要说明影响范围和建议方案
- 尊重用户的业务决策，提供专业建议但不强加观点
