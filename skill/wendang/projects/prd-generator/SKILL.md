# 需求文档生成器

## 任务目标
- 本 Skill 用于：将用户需求转化为结构化的产品需求文档（PRD）
- 能力包含：批量图片解析、需求分析与结构化、PRD文档生成
- 触发条件：用户提出编写PRD、产品需求文档、功能需求说明书的需求，或提供设计图、原型图、ZIP包需要转PRD

## 前置准备
- **批量图片处理脚本**：调用 `scripts/batch_image_processor.py` 处理ZIP包和图片文件
- **图片解析工具**：使用 `read_image` 工具逐张解析图片
- **格式规范**：参考 [references/batch-image-format-guide.md](references/batch-image-format-guide.md) 了解解析结果文件格式
- **PRD结构**：阅读 [references/prd-structure-guide.md](references/prd-structure-guide.md) 了解标准PRD结构
- **优秀样例**：参考 [references/prd-sample.md](references/prd-sample.md) 学习优秀PRD的写作风格

## 操作步骤

### 步骤1: 批量图片处理与解析（如用户上传ZIP包）

**重要**：如果用户上传ZIP包，必须先解压并逐张解析图片，将每张图片的解析结果保存到独立文件。

#### 1.1 解压ZIP包并获取图片列表
调用 `scripts/batch_image_processor.py` 处理ZIP包：

```bash
python scripts/batch_image_processor.py --zip_path <ZIP文件路径> --output_dir <输出目录>
```

**脚本输出**：
- `extracted_images/`：解压的图片文件目录
- `image_analysis_results/`：解析结果文件目录（初始为空）
- `image_list_report.json`：图片列表报告（JSON格式）

**读取图片列表报告**：
使用 `read_file` 工具读取 `image_list_report.json`，获取：
- 图片总数
- 每张图片的文件名、路径、大小
- 推断的图片类型（流程图/界面设计图/原型图）

#### 1.2 逐张解析图片
**对每张图片执行以下操作**：

1. **读取图片信息**：从图片列表报告中获取该图片的路径和推断类型
2. **选择prompt模板**：根据图片类型选择对应的prompt（见下方表格）
3. **调用 `read_image` 工具**：
   - `urls`: 图片的完整路径
   - `prompt`: 根据图片类型选择的专用prompt
4. **保存解析结果**：将解析结果写入独立文件

**针对不同图片类型的prompt模板**：

**重要约束**：所有prompt必须包含以下强制约束语句：
```
【强制约束】
1. 只描述图中实际可见的内容，禁止推断不存在的内容
2. 禁止添加"可能会"、"应该有"、"推测"等主观推断
3. 禁止扩写图中未标注的功能或业务规则
4. 准确记录文字内容，不要意译或改写
```

| 图片类型 | 推荐prompt模板 | 识别重点 |
|---------|---------------|---------|
| 流程图 | "这是业务流程图。【强制约束】只描述图中实际可见的内容，禁止推断不存在的内容。请识别：1. 流程的开始和结束节点（图中可见的元素）2. 所有流程步骤和决策点（准确记录文字）3. 步骤之间的流转关系（箭头和连线的方向）4. 图中标注的文字说明（原样记录）。只输出图中能看到的内容。" | 流程节点、决策点、连线关系、文字标注 |
| 界面设计图 | "这是产品界面设计图。【强制约束】只描述图中实际可见的内容，禁止推断不存在的内容。请识别：1. 页面上的所有UI元素（按钮、输入框、下拉框、图标等，准确记录文字内容）2. 页面的布局结构（顶部、中部、底部等区域）3. 各区域的标题和标签（原样记录文字）。只输出图中能看到的内容。" | UI组件、布局结构、文字内容、标注信息 |
| 原型图/线框图 | "这是产品原型图/线框图。【强制约束】只描述图中实际可见的内容，禁止推断不存在的内容。请识别：1. 页面包含的功能模块和区域（标注区域名称）2. 每个区域的标题和内容（准确记录文字）3. 所有交互元素（按钮、输入框、下拉框等，准确记录文字）4. 数据字段名称（列表标题）。只输出图中能看到的内容。" | 功能模块、区域标题、交互元素、字段名称 |
| 未分类 | "【强制约束】只描述图中实际可见的内容，禁止推断不存在的内容。请识别：UI元素、布局结构、文字内容、标注信息。只输出图中能看到的内容。" | 全面识别可见内容 |

**解析结果文件命名和格式**：
- 文件名：`{原图片名（不含扩展名）}_analysis.md`
- 文件路径：`image_analysis_results/{文件名}.md`
- 格式：严格按照 [references/batch-image-format-guide.md](references/batch-image-format-guide.md) 中的"解析结果文件格式"

**解析深度要求**：
**重要原则**：只描述图中实际可见的内容，禁止推断不存在的内容。

每张图片的解析应包含：
- 页面功能模块和区域划分（基于图中标注）
- 每个区域的标题和内容（准确记录文字）
- 页面跳转关系（仅限图中明确标注的）
- 所有交互元素（按钮、输入框、下拉框、链接等，准确记录文字）
- 数据字段名称（列表标题）

**禁止事项**：
- ❌ 禁止推断图中不存在的功能
- ❌ 禁止添加"可能会"、"应该有"、"推测"等主观推断
- ❌ 禁止扩写业务规则或校验逻辑（除非图中明确标注）
- ❌ 禁止意译或改写文字内容

#### 1.3 生成解析结果汇总文件
解析完所有图片后，创建汇总文件 `all_analysis_summary.md`：

参考 [references/batch-image-format-guide.md](references/batch-image-format-guide.md) 中的"汇总文件格式"，包含：
- 图片概览（总数、类型分布）
- 图片解析列表（每张图片的简要说明和详细解析链接）
- 功能模块汇总（按模块分类组织图片）
- 角色权限汇总
- 数据字段汇总
- 业务流程汇总

#### 1.4 输出解析结果供用户确认
将汇总文件内容展示给用户，并明确询问：
> "以上是所有图片的解析结果汇总。每张图片的详细解析已保存到 `image_analysis_results/` 目录中。请查看汇总内容，确认是否准确？如有需要补充或修正的内容，请告知具体图片名称和修改意见。"

**用户确认后的处理**：
- ✅ 用户确认准确 → 进入步骤2
- ❌ 用户提出修正 → 根据修正意见调整对应图片的解析结果文件，重新生成汇总文件，再次确认

**疑问澄清环节**：
如果解析过程中发现问题（如图片不清晰、内容矛盾、业务场景不明确等），应主动向用户提问。详见 [references/question-guidelines.md](references/question-guidelines.md)。

**如果用户未上传ZIP包**：
- 直接进入步骤2进行文字需求分析

### 步骤2: 需求收集与分析

智能体将通过以下方式收集和分析需求信息：

1. **读取解析结果文件**（如步骤1已完成）
   - 读取 `all_analysis_summary.md` 汇总文件
   - 根据功能模块分类，读取相关的 `*_analysis.md` 详细解析文件
   - 提取关键信息：
     - 功能模块和页面结构
     - 交互元素和数据字段
     - 业务逻辑和权限控制

2. **文字需求提取**（如用户提供文字描述）
   - 识别用户提供的产品背景、业务目标、功能诉求
   - 提取关键业务实体、角色、流程

3. **需求整合与结构化**
   - **功能模块划分**：
     - 从解析结果汇总中提取功能模块分类
     - 识别核心功能模块和辅助功能模块
     - 梳理模块间的关联关系
   - **角色权限梳理**：
     - 从解析结果的"业务逻辑推断"中提取角色信息
     - 明确各角色的权限范围
     - 梳理角色与功能的对应关系
   - **业务流程梳理**：
     - 从流程图的解析结果中提取流程逻辑
     - 识别关键决策点
     - 明确异常处理机制

### 步骤3: PRD文档生成

智能体将按照标准结构生成PRD文档：

**业务逻辑推断（核心环节）**：
在此环节，智能体需要基于图片解析结果进行业务逻辑推断和需求分析：

1. **功能逻辑推断**：
   - 基于可见的按钮、输入框等元素，推断功能用途和操作流程
   - 例如：看到「新增」按钮 → 推断"支持新增功能，点击后打开新增页面"
   - 例如：看到「查询」按钮和输入框 → 推断"支持条件筛选功能"

2. **数据流转推断**：
   - 基于列表字段和操作按钮，推断数据的增删改查流程
   - 例如：看到「编辑」「删除」按钮 → 推断"支持对单条数据的编辑和删除操作"

3. **权限控制推断**（基于业务逻辑，有疑问需询问用户）：
   - 基于业务逻辑推断权限控制规则
   - 例如：看到「删除」操作 → 推断"是否需要删除权限控制？哪些角色可以删除？"
   - 例如：看到「配置」操作 → 推断"是否需要配置权限？是否只有管理员可配置？"
   - 例如：看到敏感操作（如批量删除、权限管理）→ 推断"需要特殊权限控制"
   - **重要**：如果权限控制逻辑不明确，必须向用户提问确认

4. **交互流程推断**：
   - 基于页面元素和跳转标注，推断用户操作流程
   - 例如：看到导航栏和标签页 → 推断"用户可通过导航切换不同功能模块"

5. **校验规则推断**（允许常见校验规则推断）：
   - 基于字段类型和业务场景，推断常见的校验规则
   - 例如：看到「创建时间」字段 → 推断"系统自动生成，不可修改"
   - 例如：看到「渠道名称」字段 → 推断"必填，最大长度50字符，不允许重复"
   - 例如：看到「渠道id」字段 → 推断"必填，系统内唯一标识"

**推断原则**：
- ✅ 基于图片解析结果中的已有元素进行推断
- ✅ 推断需合理，符合常见业务场景
- ✅ 允许推断常见校验规则（如必填、长度限制、唯一性、系统自动生成等）
- ❌ 不推断图片解析结果中未提及的功能
- ❌ 不添加过度主观的假设

**疑问处理**：
- **生成前**：确认文档定位、详细程度、格式要求（详见 [references/question-guidelines.md](references/question-guidelines.md)）
- **生成中**：遇到业务价值不清晰、规则歧义、权限边界不明确等问题时，暂停并向用户提问

**处理方式**：
1. 记录所有疑问点
2. 在当前章节完成后，集中向用户提问
3. 根据用户反馈调整内容
4. 继续撰写后续章节

**文档结构**：
**重要**：每个页面及弹窗都必须按照 [references/prd-sample.md](references/prd-sample.md) 优秀样例的格式和深度来撰写。结构框架参考 [references/prd-structure-guide.md](references/prd-structure-guide.md)。

1. **文档概述**
   - 文档目的
   - 产品背景
   - 目标用户

2. **权限说明**（如适用）
   - 角色定义
   - 权限矩阵

3. **功能模块详细说明**
   **每个页面/弹窗必须包含以下内容**：
   - **需求描述**：说明功能的业务价值和用户场景
   - **页面字段**：列出页面涉及的数据字段（包含字段名称、字段说明、是否必填、校验规则）
   - **规则说明**：业务规则、权限控制、数据校验
   - **交互说明**：用户操作流程、界面交互逻辑
   - **逻辑说明**：业务逻辑、算法规则（如适用）
   
   **参考文档**：
   - 结构框架：[references/prd-structure-guide.md](references/prd-structure-guide.md)
   - 优秀样例：[references/prd-sample.md](references/prd-sample.md)

4. **非功能需求**（如适用）
   - 性能要求
   - 安全要求
   - 兼容性要求

参考 [references/prd-sample.md](references/prd-sample.md) 中的写作风格，确保文档的专业性和可读性。

### 步骤4: 专业审查与质量检查

智能体将以**产品经理**和**开发工程师**的双视角，对生成的PRD进行全面审查。

**审查维度**：
- **产品经理视角**：业务逻辑、用户体验、产品定位
- **开发工程师视角**：技术可行性、开发细节、技术架构

**审查流程**：
1. 参照 [references/prd-review-checklist.md](references/prd-review-checklist.md) 中的审查清单逐项检查
2. 将发现的问题按优先级（高/中/低）整理
3. 生成结构化的疑问清单（格式见参考文档）
4. 向用户展示疑问清单，询问处理方式

**用户确认环节**：
向用户展示审查报告后，询问：
> "PRD文档已完成专业审查，发现{数量}项疑问。请确认处理方式：[逐项确认/全部修改/全部保留]"

**疑问处理方式**：
- ✅ 确认修改 → 根据反馈调整PRD对应章节
- ✅ 保留原样 → 在疑问清单中标注"已确认"
- ✅ 稍后处理 → 记录到待办事项

**闭环确认**：
所有疑问处理完成后，询问用户：
> "是否需要重新审查？" [是/否]

### 步骤5: 最终交付

输出最终的PRD文档，并提供以下信息：
- PRD文档文件路径
- 疑问处理记录（如有）
- 审查报告（已标注用户确认状态）
- 后续建议（如需要）

  - 用途：展示优秀PRD的写作风格和深度
  - 何时读取：生成具体模块时，参考描述方式和细节深度

  - 何时参考：遇到疑问需要向用户提问时

  - 何时读取：PRD生成完成后，进行专业审查时

## 注意事项
- **图片解析原则（核心原则）**：只描述图中实际可见的内容，禁止推断不存在的内容。禁止添加"可能会"、"应该有"、"推测"等主观推断。禁止扩写业务规则或校验逻辑（除非图中明确标注）。详见 [references/image-analysis-example.md](references/image-analysis-example.md)
- **PRD写作标准（重要）**：每个页面及弹窗都必须按照 [references/prd-sample.md](references/prd-sample.md) 优秀样例的格式和深度来撰写，确保文档质量符合业界标准
- **按钮格式规范（强制）**：交互说明部分的按钮操作描述，必须使用双引号格式。正确示例：`- "编辑"：点击弹出编辑产品弹窗`，错误示例：`- **编辑**：点击弹出编辑产品弹窗`
- **主动提问原则**：遇到任何疑问必须主动向用户提问，不要猜测或假设。提问规范详见 [references/question-guidelines.md](references/question-guidelines.md)
- **专业审查机制**：PRD生成后必须进行双视角审查（产品经理+开发工程师），审查清单详见 [references/prd-review-checklist.md](references/prd-review-checklist.md)
- **批量图片处理优先**：如用户上传ZIP包，必须先解压并逐张解析，每张图片生成独立解析结果文件
- **用户确认机制**：关键环节（解析汇总、疑问处理、审查结果）必须明确询问用户确认
- **闭环处理**：审查发现的疑问必须逐一确认处理方式，支持重新审查直到问题解决

## 使用示例

### 示例1: 从ZIP包批量生成PRD（完整流程）
**用户输入**：上传了包含多张原型图的ZIP包，说明"这是我们客服系统的所有页面原型，请生成完整的PRD"

**执行过程**：
1. **步骤1：批量图片处理与解析**
   - **子步骤1.1**：调用脚本解压ZIP包
     ```bash
     python scripts/batch_image_processor.py --zip_path "./prototype.zip" --output_dir "."
     ```
     - 输出：`extracted_images/` 目录（包含10张图片）
     - 输出：`image_list_report.json`（图片列表报告）
   
   - **子步骤1.2**：读取图片列表报告
     - 使用 `read_file` 读取 `image_list_report.json`
     - 发现10张图片：渠道维护.png、员工管理.png、工作组配置.png等
   
   - **子步骤1.3**：逐张解析图片
     - 对"渠道维护.png"调用 `read_image`，使用原型图专用prompt
     - 将解析结果保存到 `image_analysis_results/渠道维护_analysis.md`
     - 对"员工管理.png"调用 `read_image`，保存解析结果
     - ...（继续处理所有图片）
   
   - **子步骤1.4**：生成汇总文件
     - 创建 `all_analysis_summary.md`
     - 按功能模块分类：网聊维护、呼叫管理、员工管理等
   
   - **子步骤1.5**：输出汇总结果并询问用户确认
     - 展示汇总内容
     - 询问："请确认解析结果是否准确？"
     - 用户确认："准确"

2. **步骤2：需求分析**
   - 读取 `all_analysis_summary.md` 获取整体情况
   - 根据功能模块分类，读取相关详细解析文件
   - 整合分析功能模块、角色权限、业务流程

3. **步骤3：生成PRD**
   - 按照标准结构生成完整的客服系统PRD
   - 包含：文档概述、权限说明、网聊维护模块、呼叫管理模块等

4. **步骤4：质量检查**
   - 检查PRD结构完整性和逻辑一致性
